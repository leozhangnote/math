# UEPG Registry Guide

## Overview

The registry is the **builder's database** ‚Äî used by Claude and the project owner to track every UEPG, prevent duplicates, and manage connections. It is never read by end users.

The registry is split into **grade-based shards** (one JSON file per grade level). This design aligns naturally with how math curricula are structured and makes connection search scope automatic and unambiguous.

---

## Shard Files

| File | Grade Coverage | Typical Content |
|------|---------------|-----------------|
| `registry_grade_4.json` | Grade 4 and below (K‚Äì4) | Counting, place value, basic operations, intro to fractions, basic geometry |
| `registry_grade_5.json` | Grade 5 | Fraction operations, decimals, intro to ratios, coordinate plane |
| `registry_grade_6.json` | Grade 6 | Ratios, percents, negative numbers, intro to algebra, area/volume |
| `registry_grade_7.json` | Grade 7 | Proportional relationships, linear expressions, equations/inequalities, probability |
| `registry_grade_8.json` | Grade 8 | Linear functions, systems of equations, exponents, Pythagorean theorem |
| `registry_grade_9.json` | Grade 9 | Quadratics, polynomials, factoring, coordinate geometry, congruence |
| `registry_grade_10.json` | Grade 10 | Polynomial functions, rational expressions, conics, intro trigonometry |
| `registry_grade_11.json` | Grade 11 | Trig functions/identities, exponential/log functions, sequences, vectors |
| `registry_grade_12.json` | Grade 12 | Limits, derivatives, integrals, matrices, probability distributions |

**Future subjects** (e.g., physics) follow the same pattern: `registry_physics_grade_9.json`, `registry_physics_grade_10.json`, etc.

**Shard splitting**: when a single shard exceeds ~500 entries, split by sub-topic within that grade (e.g., `registry_grade_9_algebra.json`, `registry_grade_9_geometry.json`).

### Grade assignment rule
Each UEPG is registered in the shard for its **target grade**. The grade-4 shard covers all concepts introduced at grade 4 or earlier.

## Entry Schema

```json
{
  "id": "string ‚Äî unique slug, matches HTML filename without extension",
  "title": "string ‚Äî human-readable display title",
  "grade": "number ‚Äî target grade level, e.g. 4",
  "subject": "string ‚Äî topic area within the grade, e.g. 'arithmetic', 'algebra', 'geometry'",
  "concept_fingerprint": "string ‚Äî canonical plain-language description for deduplication (see below)",
  "topics": ["array of keyword strings ‚Äî for grep pre-filtering and context"],
  "skills_summary": "string ‚Äî comma-separated sub-skills the UEPG covers",
  "connections": {
    "foundational": ["array of UEPG ids this page links to as prerequisites ‚Äî only built UEPGs"],
    "challenge": ["array of UEPG ids this page links to as advanced follow-ons ‚Äî only built UEPGs"]
  },
  "file": "string ‚Äî relative path from project root, e.g. 'grade_4/fraction_simplification_grade4.html'",
  "status": "string ‚Äî one of: complete | draft | retired",
  "created": "string ‚Äî ISO date, e.g. '2026-03-18'"
}
```

> **Only built UEPGs belong in the registry.** Never add an entry for a planned or unbuilt UEPG. The registry reflects what exists, not what is intended.

### The `concept_fingerprint` Field

The most important field for deduplication. Rules:
- Written in **plain mathematical language**, not title/marketing language
- Grade-level framing stripped out (that lives in `grade`)
- Specific enough to distinguish from closely related concepts
- Generated by Claude at creation time

**Good fingerprints:**
- `"reducing a fraction to lowest terms by dividing numerator and denominator by their GCD"`
- `"solving a system of two linear equations in two unknowns by substitution"`
- `"finding the derivative of a polynomial using the power rule"`

**Bad fingerprints (too vague):**
- `"fraction stuff"` ‚Äî not specific enough
- `"linear equations"` ‚Äî too broad, doesn't describe the method or scope
- `"derivatives"` ‚Äî could mean dozens of different UEPGs

## Search Scope Rules

The grade-based sharding makes search scope automatic:

| Operation | Shards to read |
|-----------|---------------|
| Deduplication check for grade-N UEPG | `grade_N` only |
| Find foundational connection candidates | `grade_{N-1}` + `grade_N` |
| Find challenge connection candidates | `grade_N` + `grade_{N+1}` |
| Batch gap analysis around grade N | `grade_{N-1}` + `grade_N` + `grade_{N+1}` |

**Edge cases:**
- Grade 4 (lowest shard): foundational search reads `grade_4` only (no grade_3 shard exists)
- Grade 12 (highest shard): challenge search reads `grade_12` only (no grade_13 shard exists)
- Concepts that span grades: read one extra shard if needed, maximum 3 shards per search

## Mandatory Workflow: Before Creating Any UEPG

> **Never create a new UEPG without completing the deduplication check first.**

### Step 1 ‚Äî Identify the target grade

Determine which grade shard the new concept belongs to.

### Step 2 ‚Äî Read the full shard

Claude reads the entire `registry_grade_N.json`. At designed scale (~300‚Äì500 entries per shard √ó ~400 bytes = ~120‚Äì200KB), this is a single fast file read.

### Step 3 ‚Äî Semantic deduplication check

Claude scans every entry's `concept_fingerprint` and `topics` for semantic overlap. This is a semantic judgment, not keyword matching ‚Äî different titles covering the same mathematical idea count as duplicates.

Claude must explicitly answer:
- **Exact duplicate** (same concept, same scope) ‚Üí Do not create. Report to owner.
- **Near-duplicate** (same concept, different grade/depth) ‚Üí Flag to owner. May be intentional.
- **Clearly distinct** ‚Üí Proceed to build.

### Step 4 ‚Äî Report findings to owner

```
Deduplication check for: [proposed title]  (Grade N)
Concept fingerprint: [generated fingerprint]
Shard checked: registry_grade_N.json ([X] entries scanned)

EXACT DUPLICATES:  none | [id: title ‚Äî explanation]
NEAR-DUPLICATES:   none | [id: title ‚Äî explanation of difference]
VERDICT: clear to build | needs owner decision
```

Only proceed after owner confirms.

### Step 5 ‚Äî Build the UEPG

Follow `templates/UEPG_GUIDE.md`.

### Step 6 ‚Äî Connection search

1. Read `registry_grade_{N-1}.json` + `registry_grade_N.json` ‚Üí identify foundational candidates
2. Read `registry_grade_N.json` + `registry_grade_{N+1}.json` ‚Üí identify challenge candidates
3. Only consider entries with `status: complete` ‚Äî never link to drafts or unbuilt pages
4. Present candidates to owner (format in `UEPG_GUIDE.md` Step 7)
5. After owner confirmation: wire connections in both HTML files and both registry entries immediately

### Step 7 ‚Äî Register the new UEPG

Append the new entry to `registry_grade_N.json`. Increment `_entry_count`. Update `_last_updated`.

The entry `connections` arrays must only contain ids of UEPGs that are already built and registered with `status: complete`.

## Workflow: Batch Planning / Gap Analysis

When asked "suggest what to build next around concept X at grade N":

1. Read `registry_grade_{N-1}.json` + `registry_grade_N.json` + `registry_grade_{N+1}.json`
2. Map the concept neighborhood ‚Äî what exists, what's missing
3. Present a gap analysis to owner:

```
Gap analysis around: [concept X]  (Grade N)
Shards read: grade_{N-1} ([X] entries), grade_N ([Y] entries), grade_{N+1} ([Z] entries)

ALREADY BUILT:
  - [id]: [title] (grade N, status: complete)

GAPS ‚Äî suggested new UEPGs to build:
  1. [proposed title]  (Grade N)
     Fingerprint: [fingerprint]
     Reason: [why this is a natural gap]

  2. ...

Suggested build order: [dependency order]
Please confirm which to build and in what order.
```

## Keeping Connections Consistent

The registry is the **source of truth** for connections. HTML files reflect the registry, not the other way around.

Rules:
- **Only built UEPGs are linked** ‚Äî connections always point to entries with `status: complete`
- Every connection in an HTML file must exist in the registry as a `complete` entry
- Every registry connection must be reflected in both HTML files (both directions)
- When a new UEPG is built and a connection is confirmed, both sides are updated immediately in the same session ‚Äî no deferred updates

---

## Registry Maintenance

- **After every build session**: commit updated shard JSON(s) to git
- **Never add unbuilt entries**: the registry only reflects what has been built
- **Retiring pages**: if a UEPG is superseded or removed, set `status: "retired"` to preserve connection history ‚Äî never delete entries
- **Shard splitting**: at ~500 entries, split by sub-topic within grade (e.g., `registry_grade_9_algebra.json`)
- **`_entry_count`**: update manually when adding entries ‚Äî used for quick size checks without parsing the full array
